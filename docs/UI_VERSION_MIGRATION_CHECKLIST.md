## 2. 迁移边界与约束

- **业务边界**：登录、歌单、搜索、播放、歌词继续由 React Store/Service 提供；UI 壳层只通过 `window.__ALLMUSIC_BRIDGE__` 消费。
- **一致性边界**：不同 UI 允许视觉差异，但功能行为（播放、切歌、进度、音量）必须一致。
- **性能边界**：iframe UI 定时同步播放器状态，避免过高频率调用。

### 2.1 Bridge 中心缓存策略（方案 B，已启用）

- **缓存对象**：`loadPlaylists`、`loadPlaylistDetail`。
- **SWR 策略**：先返回缓存（秒开）→ 后台刷新 → 通过 `allmusic:bridge-cache-updated` 通知当前 UI 静默更新。
- **缓存时效**：歌单 `60s`、详情 `30s`；超出 fresh TTL 但未超过最大陈旧窗口时可先用旧数据。
- **请求去重**：同一 `cacheKey` 的并发请求复用同一个 Promise，避免切换 UI 时请求风暴。
- **鉴权隔离**：缓存 key 绑定 auth fingerprint（用户 + cookie 哈希），登录态变化后自动隔离旧缓存。

## 3. 风险清单（红黄绿）

- 🔴 **高风险**：V9 沉浸详情未完全接入，当前仍以主列表壳层为主。
- 🟡 **中风险**：歌词来源质量受接口返回影响，部分歌曲可能仅显示“暂无歌词”。
- 🟡 **中风险**：跨 UI 切换时存在毫秒级时间漂移（已通过快照恢复降低影响）。
- 🟢 **低风险**：UI 版本状态持久化与旧主题值迁移已覆盖。

## 4. 测试建议（最小集）

1. **切换连续性**：Current 播放中依次切到 V4 / V8 Clay / V8 Fallout，再切回 Current，检查是否不断播、时间是否连续。
2. **功能一致性**：在 V8 Clay 与 V8 Fallout 中分别验证“歌单播放 / 搜索播放 / 上下曲 / 拖动进度 / 调音量”。
3. **登录依赖**：仅登录单平台时进入 V8 页面，确认状态提示清晰且不会崩溃。
4. **本地 API 重连**：触发 `allmusic:local-api-ready` 后，V8 页面可重新拉取歌单。
5. **缓存体验**：在 V4/V8 间连续切换 5 次，观察歌单区域是否能直接秒开且无明显“从空到有”重载闪烁。
