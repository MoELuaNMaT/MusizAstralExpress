- 白色功能键的按压反馈统一收敛为“厚度变化”而不是“整体下移”：默认态增加底边厚度，激活态保留基础厚度但明显变薄，按钮底边与上方灯位始终保持对齐不动。
- 白色功能键的点亮演出进一步收敛为分段时序：`pressing(8→1)`、`hold(1 停顿)`、`rebound(点亮后 1→3)`、`settled(稳态 3)`。灯和图标只在 `rebound` 阶段开始时点亮，不和按下动作同步触发。
- 白色功能键的时序微调为 `hold=380ms`、更快回弹；结构层级也同步调整为“灯 -> 白色标签 -> 按钮面”，按钮面只保留放大的功能图标，文字不再进入按钮面。
- 音量旋钮外圈不再使用完整环槽，统一收敛为与 `KNOB_MIN_ANGLE ~ KNOB_MAX_ANGLE` 对应的 270 度弧形槽；不可旋转的底部区间直接留空，避免错误的 360 度语义。
- 为避免影响字幕旋钮，音量旋钮单独增加 `am-retro-player__knob-shell--volume` 修饰类；只有 volume shell 被改成 270 度弧形槽，公共 `knob-shell` 结构继续保留给其他旋钮复用。
- 音量旋钮的可旋转弧槽进一步细化为 10 格扇形刻度，并沿弧线从左到右分配为 `3 绿 / 4 黄 / 3 红`。点亮规则统一为按当前音量百分比每 `10%` 激活一格，未激活段保留各自的暗色底色。
- 音量刻度段的几何实现统一收敛为“局部角度 + 统一起始角”：每格在 270 度总弧段内按 `27deg` 步长划分，并从 `225deg` 起始角展开。槽带厚度同步加宽，确保环带宽度和扇形刻度的视觉尺度一致。
- 音量刻度的层级关系进一步收敛为“槽体优先、分段嵌入”：`knob-shell--volume::before` 负责绘制完整 270 度黑色厚槽，`knob-scale` 再以内缩环带方式承载 10 个彩色扇形格，确保用户先看到槽，再看到嵌入槽内的分段。
- 音量刻度的最终实现从 CSS mask 切换为 SVG：`knob-scale-svg` 绘制一条 270 度黑色槽体路径和 10 条内缩扇形路径。这样在当前小尺寸 UI 下更稳定，也更容易精确控制槽宽、段宽和层级。
- 音量槽与分段的半径关系进一步收敛为“槽更大、段更厚”：槽体使用更小的内半径和更大的外半径，扇形段使用略内缩但明显更厚的环带，优先保证“宽厚扇形块嵌在大槽里”的视觉关系。
- 播放进度条的版式收敛为“框内时间 + 中间刻度槽 + 针脚只跑刻度区”：时间读数置于框内两端，删除 `0~100` 数字刻度，红针按固定左右内边距在第一根与最后一根刻度线之间移动；整条进度框整体上提，与音量旋钮弧槽顶点齐平。
- 进度框的最终细化收敛为“少量上提 + 上刻度下时间”：`dial-wrap` 只轻微负 margin 上移，避免高过音量弧槽；`dial` 内部顺序固定为先刻度槽和红针，再显示底部时间读数。
- 进度指针的视觉语义进一步收敛为“表头式底部锚定针 + 槽内暖光柱”：去掉顶部三角针头，用底部齐平的细圆柱红针表示当前时间，并在其上方叠加一层位于槽内的柔和橙黄渐变光柱，光照高度约覆盖槽内下方三分之二区域。
- 进度指针层级最终固定为 `刻度(z=1) -> 红针(z=2) -> 槽内暖光(z=3)`；暖光不再跟随指针位置，只作为整条进度槽的底部背光存在，红针单独按进度值横向移动并持续锚定在槽底。
- 进度背光的最终实现放弃雾状 blur，改为整槽纵向渐变光幕；同时在 `dial` 底部增加固定遮罩，让红针实际延伸到槽底以下但视觉上被底板盖住，确保“从底部伸出”的机械关系成立。
- 进度条最终采用“外框 + 内部槽体 + 槽内元素”的三层结构：`dial-track` 负责定义刻度槽的几何边界与裁切，刻度、暖光和红针全部绑定到该槽体内部；这样背光会对齐整条槽，红针也能通过槽体裁切实现“底部插入”的效果。
- 进度条的 `dial-track` 最终语义固定为“整块内凹深色槽体”，而不是单独的刻度带；因此时间读数也保留在 `dial-track` 内部底部，两端对齐，确保暖光、红针、刻度和时间共享同一套槽体边界。
- 进度槽底部遮罩最终只保留薄薄一层裁切唇边，避免再次把贴底的暖光和红针抬离槽底；槽底的“贴合感”由 `lamp/needle` 的 `bottom: 0` 和轻量裁切共同实现，而不是靠厚遮罩伪造。
- 进度槽的最终裁切责任继续收敛到 `dial-track` 自身：底部不再叠额外暗遮罩层，暖光用整槽高度的纵向渐变在下方三分之二区域发亮，红针则向槽底以下延伸后由 `overflow: hidden` 自然截断，确保视觉上真正贴底。
- 当前双槽关系继续保留：红针和刻度仍然服务于内层槽视觉，但底部暖光明确收敛为外层槽的背景层，使用有效的纵向渐变并保持 `z-index: 0`，避免再次遮住红针或被误认为内层槽发光。
- 进度条最终显式拆成“双槽”结构：`dial-track` 作为外层深色槽，负责底部背光和时间；新增 `dial-scale-slot` 作为内层窄槽，专门承载刻度与红针。只有这样，外层背光和内层红针才拥有独立的定位与裁切上下文。
- 进度条的正确两层结构修正为：`dial` 才是外层黑槽，负责底部背光；`dial-track` 是中层内容槽，负责刻度、红针和时间。新增第三层显式内槽会制造不存在的视觉层级，因此已撤回。
- 外层背光想要可见，不仅要挂在 `dial`，还必须给 `dial` 底部腾出一条不被 `dial-track` 覆盖的可见区域。因此当前采用“外层背光 + 中层槽上提”的方式：`dial-track` 底部 inset 增大，背光停留在 `dial` 底部可见带内。
- 外层背光的几何边界最终对齐 `dial` 内边界，而不是沿用 `dial-track` 的左右 inset；否则即便层级正确，视觉上仍会被误读成中层槽的底边灯。
- 当最外层黑槽被确认是最终参照系后，进度条的最终层级修正为：`dial-track(z=1)` 承载刻度和时间，`dial-needle(z=2)` 直接挂在 `dial` 上并使用刻度区的横向对齐公式，`dial-lamp(z=3)` 作为外层槽背光覆盖在刻度和红针之上。
- 当前外层槽下的视觉比例固定为：`dial-lamp` 高度约占外层槽的一半，`dial-needle` 可见长度约占外层槽的三分之二；后续仅在这两个比例附近微调，不再改结构。
- 当前结构继续保留两层语义，但只保留一层视觉：`dial-track` 退化为透明定位容器，不再提供独立背景、阴影或圆角，从视觉上抹掉“第二层槽”的存在感。
- 外层背光的动态效果最终收敛为“慢速呼吸 + 低幅度扫光”；时间读数则下移并改成青色发光字，保持与 VFD 屏的电子荧光语言一致。
- 大刻度槽与左侧音量旋钮的垂直对齐统一在 `dial-wrap` 层完成：通过整体下移 `dial-wrap` 调整两者中线关系，不再通过改动槽体内部的 `top/bottom` 去间接凑对齐。
- 外层背光的顶部边缘通过 `dial-lamp::before` 补一层向上扩散的模糊柔光来软化；不再依赖单一主渐变层承担全部边缘过渡。
- 外层背光的扫光动画最终采用“同一轮关键帧内往返归位”的方式，避免在循环边界硬切；时间读数继续下压，使其更贴近槽底。
- 音量旋钮下方的读数最终收敛为单行大数字：移除 `VOL` 标签，只保留 `百分比 / MUTE`，并放大到接近原两行文字的总高度。
- 音量读数的最终显示格式进一步固定为两位数字数码管风格：移除 `%`，显示范围限制为 `00~99`，满值 `100` 视觉上也显示为 `99`，并略微上移读数位置。
- 音量数字的垂直对齐最终继续通过 `knob-caption` 的位移控制：当前将读数整体抬入上方弧槽空缺区，而不再保留在旋钮下方的传统标题位。
- 右下角平台开关的状态源最终收敛为“认证态 + 本地服务态”双条件：新增共享 `local-api-status` store 同步 `App` 中的 `localApiProgress.serviceState`，播放器用 `users + cookies + serviceState === ready` 决定 QQ/网易云开关是否点亮并拨到 on。

## 2026-04-11（真实频谱演出参数）
- 当前 REAL 频谱的列数固定收敛为 `20`，不继续维持旧 `16` 列。原因是当前 VFD 侧栏宽度还能承受 20 列，足以明显提升播放器感，但不会像 24 列那样压得过细。
- 频带映射收敛为 `Mel/对数感知分带 + 分区视觉增益`，而不是简单平均 FFT bin。原因是这样能自然抬升低频和中频存在感，更接近人耳和传统播放器频谱的观感。
- REAL / SIM 两种模式统一复用同一套 20 列分段柱体和峰值帽 DOM；两者只切数据来源，不再允许切模式时改变布局密度。

## 2026-04-11（真实频谱波动空间策略）
- 当前频谱的“更带劲”策略收敛为“每频带独立动态窗口 + 低频顶部软压缩 + 正向变化增强”，不继续通过全局增益或进一步加大 `pow` 指数硬推亮度。
- 每个频带的最低/最高参考值都按自己的近期能量历史自适应，低频允许更高的 floor 跟随比例和更大的 headroom，中高频则保留更灵敏的细节空间。
- 显示层当前明确引入“变化量”信息：最终柱高不是纯当前值，而是 `当前能量` 与 `上升变化` 的混合，从机制上提高动态丰富度。

## 2026-04-11（真实频谱拖尾策略）
- 频谱柱体的最终光效语义收敛为“单列亮头 + 暗尾拖尾”，而不是整列同亮度点亮。理由：这样更像播放器特效，也更容易把“当前状态”视觉焦点压在柱头上。
- 峰值帽保留，但回落速度正式减半。理由：帽子现在承担的是“上一波冲顶记忆”，更慢的回收更符合这轮想要的轻飘拖尾感。
- 显示值在进入 attack/release 前增加一次分段区间重映射：底部低值区压缩，中段高敏感区扩展，顶部只保留少量冲顶空间，避免再次回到“低频常顶死、整体只抖几格”。

## 2026-04-11（真实频谱主体动态关系）
- 主体柱体的时间关系正式收敛为：`上升速度 > 主体回落速度 > peak hold 回落速度`。本轮通过放慢主体 release，而不再继续改 attack 或 peak hold，确保这三个层次更清晰。
- 为了让高潮比前奏更容易冲高，当前不再优先保持“长期稳定”，而是明确偏向“高潮可冲、前奏可收”。因此增益、headroom 和最终 remap 都按更高动态对比重新校准。

## 2026-04-11（真实频谱成熟算法路径）
- 真实频谱的后续增强路径正式从“继续纯手调映射曲线”收敛为“前端近似 PCEN + spectral flux 混合”。原因是用户当前主要不满的是“除强节奏外其他内容不够活、高潮不够冲”，这本质上属于响度自适应和变化检测问题，不再是单纯 remap 参数问题。
- 最终显示值当前固定由三部分组成：`absoluteEnergy` 保留真实响度基线，`PCEN-like energy` 负责把相对近期平均值抬起来，`fluxEnergy` 负责强调正在发生的变化。这样既不把频谱做成假动画，也避免持续性内容长期贴在中腰线。
- 低频治理策略继续保持“先打开动态窗口，再做轻度软压缩”，不退回单纯加大低频压缩的方案。原因是用户目标是给左侧腾出波动空间，而不是把左侧永久压矮。

## 2026-04-11（频谱色阶重新分配）

## 2026-04-15（安装版 vendor 缓存必须做资源指纹校验）
- 安装版运行时缓存的判定标准正式从“脚本文件存在即视为已解压”收敛为“关键运行时文件完整 + 资源包指纹一致”。
- 原因：项目当前存在“同版本号下重复重打包”的工作流，只靠 `version` 或“目录存在”无法判断 `vendor.zip` 是否已经更新；继续沿用旧缓存会直接把新安装包退化成旧运行时。
- 当前实现改为：
  - 读取安装资源目录中 `vendor.zip` 的元数据，生成资源指纹。
  - 校验 `vendor/.vendor-stamp` 与关键文件：`scripts/start-netease-api.cjs`、`scripts/start-qmusic-adapter.cjs`、`runtime/node/node.exe`、`runtime/qq-adapter/ALLMusicQQAdapter.exe`、`node_modules/NeteaseCloudMusicApi/app.js`。
- 任一项缺失或指纹不一致时，删除旧 `vendor` 并强制重新解压，再写入新的 `.vendor-stamp`。
- 这次修复不引入兼容分支，也不依赖用户手动删缓存；安装版启动时应自行完成运行时升级。

## 2026-04-15（本地 API 就绪标准与前端探活策略收敛）
- 桌面端本地 API 的“已就绪”标准正式收敛为“HTTP 健康接口返回 200”，不再接受“仅端口可连接”作为成功条件。
- 原因：QQ 适配器进程与端口会先起来，但 `FastAPI/Uvicorn` 的实际路由在短暂窗口内仍可能不可用；把这一阶段误判为 ready，会直接导致登录点击时 `fetch failed`，并诱发前端健康检查把同一服务反复判成“已就绪/异常”。
- 与此配套，前端健康检查 effect 正式改为：
  - 单实例常驻轮询，不再依赖 `serviceState` 变化重建 effect。
  - 对失败服务先做一次短延迟复探，再决定是否进入自动恢复。
- 这样桌面端的启动、自检、自动恢复三套状态判断都以同一组 HTTP 健康标准收敛，避免 UI 状态与真实服务可用性再次分叉。

## 2026-04-15（QQ 扫码登录前必须主动等待本地服务就绪）
- QQ 扫码登录链路的进入条件正式收敛为“弹窗打开后再次等待 `3001/health` 可用”，而不是只信任外层 `isLocalApiReady` 的瞬时状态。
- 原因：用户点击 QQ 登录的时机可能落在 QQ 适配器仍在 warm-up 的几秒窗口内；即便外层状态刚被判成 ready，`/connect/qr/key` 仍可能首发失败。
- 当前实现放在 `auth.service.ts -> qqQRCodeLogin()` 内部，而不是散落到 UI 组件中：
  - 统一在请求二维码 key 前轮询 `/health`。
  - 等待期间通过 `onStatusChange` 向扫码弹窗回传“正在等待 QQ 本地服务就绪...”。
  - 超时后仍按真实错误返回“无法连接 QQ 接口”，不做静默兜底。
- 这样能保证所有调用 QQ 扫码登录的入口共享同一条就绪策略，不再把启动竞态暴露给具体 UI 组件。

## 2026-04-15（安装版随包资源目录改为按指纹分桶）
- 安装版随包运行时目录正式从固定 `AppData/.../vendor` 改为 `AppData/.../vendor/bundle_<resource_stamp>`。
- 理由：固定目录方案要求先删除旧 `vendor` 再重解压，一旦旧 QQ 适配器或本地 API 子进程仍占用旧目录，资源刷新就会失败；继续把错误吞掉还会误导成“项目根目录不存在”。
- 新策略下：
  - 每份 `vendor.zip` 按自身资源指纹落到独立子目录。
  - 新版本启动时不需要删除旧 bundle，因而不会被历史子进程锁住。
  - 当前会话的 `project_root` 直接指向新的 bundle 目录，不再依赖固定根目录刷新成功。
- 与此配套，安装版的 `ensure_vendor_extracted()` 失败必须直接上抛，不能在命令层继续 `.ok()` 吞掉；只有开发模式下明确不存在 `vendor.zip` 时，才允许回退到工作区解析。

## 2026-04-15（QQ 服务就绪标准必须包含“扫码路由存在”）
- 桌面端对 QQ 本地服务的“已就绪”定义正式从“`/health` 返回 200”收紧为“两项同时成立”：
  - `http://127.0.0.1:3001/health` 返回 `200`
  - `http://127.0.0.1:3001/connect/qr/create` 不是 `404`
- 设计理由：
  - 这次实机根因已经证实，旧版 Node 服务也会返回相同的 `/health`，但不具备扫码路由。
  - 只看健康页会把“占着 3001 的旧服务/开发服务”误判成当前安装包自带 QQ API，进而导致自检正常、扫码失败两套状态分叉。
  - 对 `/connect/qr/create` 故意不传 `key`，让 FastAPI 直接返回 `422`；这样探测的是“路由是否存在”，而不是“二维码是否真生成成功”，不会把上游网络抖动混进服务就绪标准。
- 该标准必须同时应用在三条链路：
  - Rust 后端 `ensure_local_api_services_inner()` 的 readiness 判断
  - 前端 `App.tsx` 的桌面端健康检查与自动恢复
  - 前端 `auth.service.ts` 的 QQ 扫码登录前等待逻辑
- 配套策略：如果桌面端尝试启动 QQ 服务后，管理的子进程在 ready 前立即退出，应直接 fail-fast 报端口占用/版本不兼容，而不是继续等待统一超时。
